# 数据收发

TCP 通信是全双工的，收取数据和发送数据是独立的，所以利用同一个 fd 收取和发送数据不会互相影响

不建议多个线程同时利用一个 fd 收取数据或发送数据，因为 TCP 数据是有序的，如果多个线程同时读取或发送，则无法保证数据的顺序

可以对同一个 fd，在一个线程收取数据，在另一个线程发送数据，但这样需要同步两个线程的 fd 的错误状态

## 收取数据

1. 在接受连接并获取客户端 fd 后，会将 fd 绑定到相应的 I/O 复用函数上并监听可读事件

2. 可读事件触发后，调用 `recv` 从 fd 收取数据，根据不同业务需求，可能会收取部分数据或一次性收完所有数据

3. 将收取的数据放入缓冲区，然后做解包操作

收完数据的标志是 `recv` 返回 -1，错误码为 `EWOULDBLOCK` 或 `EAGAIN`

## 发送数据

非 `epoll` 的 ET 模式：

1. 一开始不注册 fd 的可写事件，因为只要对端收取数据正常，一般不会出现因 TCP 窗口太小导致本端 `send` 失败，而注册了可写事件后会一直触发

2. 如果有数据需要发送时，先调用 `send` 直接发送，如果在发送过程中函数返回 -1，并且错误码是 `EWOULDBLOCK` 或 `EAGAIN`，表示 TCP 窗口太小无法发送数据；如果还有数据需要发送，可以给 fd 注册可写事件，并将数据存入自定义的缓冲区中，等到写事件触发后再发送缓冲区的数据，直到所有数据发送完毕

3. fd 注册可写事件后，在写事件触发并发送数据后仍有数据未能发送完成，就不用再次注册可写事件，上次注册的可写事件继续有效

`epoll` ET 模式：

1. 注册写事件并触发写事件后，发送数据但未能全部发送，需要再次注册写事件，以便让写事件能够继续触发

对于一段数据反复发送都不能完全发送完，需要设置一个最大发送次数或最大发送总时间，超过限制后可以认为对端出了问题，应该立即清空缓冲区并关闭相应连接
